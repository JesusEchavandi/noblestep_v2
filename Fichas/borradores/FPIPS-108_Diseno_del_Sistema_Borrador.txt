Proyecto: Plataforma Web y Ecommerce para la gestion y venta de calzado de Noble Step S.A.C.
Version: 1.0
Fecha: 14/04/2026

1. HISTORIAL DEL DOCUMENTO
- Version: 1.0
- Fecha: 14/04/2026
- Responsable: Equipo NobleStep
- Observacion: Borrador integral de diseno funcional y tecnico.

2. INTRODUCCION
Documento de diseno que describe interfaces clave, arquitectura, persistencia y diagramas tecnicos del sistema NobleStep.

3. OBJETIVOS
- Definir la estructura tecnica implementada.
- Alinear interfaces de usuario con casos de uso.
- Documentar despliegue y relaciones entre componentes.

4. ALCANCES
Incluye:
- Diseno de interfaces web (ecommerce y admin).
- Modelo de persistencia (ER fisico).
- Vista de componentes, despliegue, secuencia y clases.
No incluye:
- Diseno de app movil nativa.

5. DISENO (ECU) Y PROPOSITO
5.1 Interfaz Ecommerce - Catalogo
- CU asociado: Explorar catalogo.
- Pantalla muestra: listado de productos, filtros por categoria, detalle de variante, precio y stock.

5.2 Interfaz Ecommerce - Carrito/Checkout
- CU asociado: Confirmar pedido.
- Pantalla muestra: resumen de productos, cantidades, subtotal, total y accion de confirmar.

5.3 Interfaz Ecommerce - Login/Recuperacion
- CU asociado: Iniciar sesion / Recuperar contrasena.
- Pantalla muestra: formulario de acceso y solicitud de recuperacion por correo.

5.4 Interfaz Admin - Productos y Variantes
- CU asociado: Gestionar productos.
- Pantalla muestra: CRUD de catalogo, stock por variante, estado y filtros.

5.5 Interfaz Admin - Pedidos
- CU asociado: Gestionar pedidos.
- Pantalla muestra: lista de pedidos, detalle, cambio de estado y trazabilidad.

5.6 Interfaz Admin - Reportes
- CU asociado: Consultar reportes.
- Pantalla muestra: indicadores de ventas, periodos y resumenes.

6. MODELO DE PERSISTENCIA
6.1 Diagrama Entidad-Relacion Fisico (resumen)
- Motor: MySQL
- Entidades principales: usuarios, clientes, categorias, productos, variantes_producto, pedidos, detalle_pedido, ventas, detalle_venta, proveedores, compras, detalle_compra, audit_logs.
- Relaciones:
  - categorias -> productos (1:N)
  - productos -> variantes_producto (1:N)
  - clientes -> pedidos (1:N)
  - pedidos -> detalle_pedido (1:N)
  - ventas -> detalle_venta (1:N)
  - proveedores -> compras (1:N)
  - compras -> detalle_compra (1:N)

6.2 Relaciones de modulos
- Frontend Ecommerce consume API para autenticacion, catalogo, carrito y pedidos.
- Frontend Admin consume API para CRUD, inventario, pedidos y reportes.
- API accede a MySQL mediante capa de datos.
- API integra servicio de correo para recuperacion de contrasena.

6.3 Diagrama de clases (resumen textual)
- Modelos: Producto, VarianteProducto, Pedido, DetallePedido, Cliente, Usuario, Venta, Compra, Proveedor.
- Servicios: AuthService, TokenService, EmailService, AuditService.
- Controladores: AutenticacionEcommerceController, PedidosController, ProductosController, UsuariosController, etc.

6.4 Diagrama de despliegue (resumen textual)
Nodo VPS Linux:
- Nginx (reverse proxy)
- Contenedor Backend .NET
- Contenedor Frontend Admin
- Contenedor Frontend Ecommerce
- Contenedor MySQL
Conectividad:
- HTTPS externo -> Nginx -> servicios internos.

6.5 Diagrama de secuencia (CU Confirmar pedido)
1) Cliente envia confirmacion.
2) Frontend llama API.
3) API valida stock y reglas.
4) API guarda pedido + detalle.
5) API actualiza stock.
6) API responde confirmacion.
7) (Opcional) API dispara notificacion por correo.

6.6 Diagrama de componentes (resumen)
- Componente UI Ecommerce (Angular)
- Componente UI Admin (Angular)
- Componente API (ASP.NET Core)
- Componente DB (MySQL)
- Componente Notificaciones (correo)
