FPIPS-106 - MODELO DE ANALISIS (BORRADOR COMPLETO)
Proyecto: Plataforma Web y Ecommerce para la gestion y venta de calzado de Noble Step S.A.C.
Version: 1.0
Fecha: 14/04/2026

1. PROPOSITO
Detallar el comportamiento funcional y no funcional del sistema NobleStep, incluyendo reglas de negocio, mensajes y flujo de casos de uso criticos.

2. ALCANCE
- Aplicacion Web Ecommerce (cliente final).
- Aplicacion Web Administrativa (admin/operador).
- API backend y persistencia MySQL.

3. OBJETIVOS DEL PROYECTO
- Digitalizar proceso de venta y control de inventario.
- Reducir errores por gestion manual.
- Aumentar capacidad de atencion y alcance comercial.

4. REQUERIMIENTOS FUNCIONALES
4.1 Requerimientos funcionales (Ecommerce)
- RF-001: Registro e inicio de sesion de cliente.
- RF-002: Recuperacion de contrasena por correo.
- RF-003: Visualizacion de catalogo y variantes.
- RF-004: Gestion de carrito y confirmacion de pedido.

4.2 Requerimientos funcionales (Administrativo)
- RF-005: Gestion de productos, categorias y variantes.
- RF-006: Gestion de inventario y compras.
- RF-007: Gestion de pedidos y estados.
- RF-008: Gestion de usuarios y roles.
- RF-009: Consulta de reportes.

5. INVENTARIO DE CASOS DE USO
- CU-01 Iniciar sesion cliente (RF-001)
- CU-02 Recuperar contrasena (RF-002)
- CU-03 Confirmar pedido (RF-004)
- CU-04 Gestionar productos (RF-005)
- CU-05 Actualizar stock por compra/venta (RF-006)
- CU-06 Gestionar estado de pedido (RF-007)
- CU-07 Administrar usuarios y roles (RF-008)
- CU-08 Consultar reportes (RF-009)

6. ESPECIFICACION DETALLADA (CU CRITICO)
6.1 Caso de Uso: CU-03 Confirmar pedido
- Actor principal: Cliente
- Actores secundarios: API, Base de datos, Servicio de correo
- Precondiciones:
  - Cliente autenticado.
  - Carrito con al menos 1 item.
  - Stock disponible para todas las lineas.
- Flujo principal:
  1) Cliente confirma compra.
  2) Sistema valida stock y datos de envio.
  3) Sistema registra Pedido y DetallePedido.
  4) Sistema descuenta stock.
  5) Sistema devuelve confirmacion y dispara notificacion.
- Postcondiciones validas:
  - Pedido registrado con identificador unico.
  - Inventario actualizado.
- Postcondiciones invalidas:
  - Pedido rechazado por stock insuficiente.
  - No se persisten cambios parciales.
- Excepciones:
  - EX-01 Stock insuficiente en una o mas lineas.
  - EX-02 Error de persistencia.

7. REGLAS DE NEGOCIO
- RN-001: No confirmar pedido sin stock suficiente.
- RN-002: Operacion de pedido y stock debe ser consistente.
- RN-003: Solo admin gestiona usuarios/roles.
- RN-004: Token de recuperacion tiene expiracion.
- RN-005: Cambios criticos generan registro de auditoria.

8. MENSAJES
8.1 Mensajes de exito
- MSG-OK-001: Inicio de sesion exitoso.
- MSG-OK-002: Pedido confirmado correctamente.
- MSG-OK-003: Correo de recuperacion enviado.

8.2 Mensajes de error
- MSG-ER-001: Credenciales invalidas.
- MSG-ER-002: Stock insuficiente para completar pedido.
- MSG-ER-003: Token invalido o expirado.
- MSG-ER-004: Acceso denegado por rol.

9. REQUERIMIENTOS NO FUNCIONALES
- RNF-001 (Seguridad): JWT para autenticacion y autorizacion por roles.
- RNF-002 (Seguridad): Hash seguro de contrasenas.
- RNF-003 (Disponibilidad): Servicio web 24/7 con monitoreo basico.
- RNF-004 (Rendimiento): Respuesta de operaciones comunes < 2 segundos en carga normal.
- RNF-005 (Mantenibilidad): Arquitectura por capas y convenciones de codigo.
- RNF-006 (Trazabilidad): Registro de eventos de auditoria.

10. ATRIBUTOS DE CALIDAD
- Desempeno: consultas y operaciones optimizadas.
- Confiabilidad: manejo de errores y validaciones.
- Disponibilidad: despliegue en contenedores con reinicio automatico.
- Escalabilidad: posibilidad de escalar servicios web.
- Seguridad: politicas de autenticacion y control de acceso.

Fin del borrador FPIPS-106.
